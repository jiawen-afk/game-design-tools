# 视频导出尾部黑屏修复设计

## 问题与证据

视频处理工作台在保留 Vorbis 音频时，部分 OGV 文件播放结束前会出现约一帧黑屏。对实际文件的 FFprobe 检查结果如下：

- 源视频流：451 帧，时长约 `15.016667` 秒。
- 源音频流及容器：时长约 `15.070000` 秒。
- 输出 Theora 视频流：451 帧，时长约 `15.016815` 秒。
- 输出 Vorbis 音频流及容器：时长约 `15.069751` 秒。
- 输出尾部视频帧并非黑帧；黑屏发生在视频流结束后、音频和容器仍继续的约 `0.053` 秒。
- 静音导出没有较长音轨，因此不出现该问题。

根因不是 Upscayl 多生成图片帧，而是编码命令没有以视频流长度限制最终容器。Godot 在 Theora 视频流结束但 Ogg/Vorbis 尚未结束的空窗中显示黑屏。

## 目标

- 有音频和静音输出均以完整视频流为时间基准结束。
- 过长音频被裁切到视频结束点。
- 音频短于视频时不提前截断视频。
- 普通处理、GPU 超分、质量模式和目标大小双遍编码使用相同规则。
- 发布前验证能阻止仍有明显音频/容器尾巴的输出。

## 方案

### 输出时长基准

为视频探测结果增加明确的视频流时长，不改变现有容器时长字段的含义：

1. 优先使用 FFprobe 视频流的 `duration`。
2. 视频流未声明时，使用 `nb_frames / averageFps`。
3. 两者均不可用时，回退到容器 `duration`。

普通处理使用该视频流时长作为输出上限。

GPU 超分先提取实际帧序列，因此编码时使用 `实际提取帧数 / 目标帧率` 作为输出上限。这比源容器时长更准确，也能覆盖可变帧率源文件和帧率转换的舍入差异。

### FFmpeg 编码

`buildTheoraEncodeArgs` 接收可选的 `outputDurationSeconds`。当值有效且大于零时，在输出参数中增加：

```text
-t <outputDurationSeconds>
```

该限制同时作用于视频和音频输出：

- 音频较长时会在视频结束点裁切。
- 音频较短时，视频仍编码到指定视频时长，不会像 `-shortest` 那样被提前截断。
- 双遍编码的两遍使用同一个上限，避免码率统计与最终输出不一致。

不采用 `-shortest`，也不增加编码后的第二次 remux。

### 输出验证

`verifyGodotOgvProbe` 读取视频流、音频流和容器时长。在视频时长可用时，比较容器/音频结束点与视频结束点：

- 容许 Theora/Ogg 和音频包时间基准造成的小幅舍入差异。
- 容差使用 `max(0.05 秒, 一帧时长 + 0.01 秒)`。
- 超过容差时拒绝发布，并报告视频、音频、容器时长和允许差值。
- 静音输出同样检查容器不得明显长于视频流。
- 缺少可靠视频时长元数据时不凭空推断，不新增误阻断。

## 代码边界

- `electron/videoProcessingCommands.cjs`：探测映射、编码参数和 OGV 时长验证等纯规则。
- `electron/videoProcessingJobs.cjs`：从普通视频流或 GPU 实际帧数选择 `outputDurationSeconds`，并传给所有编码遍次和目标大小重试。
- `src/components/VideoProcessingWorkspace/videoProcessingModel.ts`：只扩展视频探测类型，不加入 FFmpeg 规则。
- UI、队列生命周期和 Upscayl 批处理逻辑不改变。

## 测试

- `videoProcessingCommands.test.cjs`：
  - 视频流时长优先于较长的容器/音频时长。
  - 视频时长缺失时按帧数与帧率推导。
  - 编码参数包含精确的 `-t`，但不包含 `-shortest`。
  - 普通、图片序列、静音和双遍参数保持现有映射。
  - 验证拒绝约 `0.053` 秒的实际尾巴，并接受容差内舍入。
- `videoProcessingJobs.test.cjs`：
  - 普通处理传入视频流时长。
  - GPU 超分以实际帧数除以目标帧率计算时长。
  - 目标大小重试继续使用同一时长上限。
- 完整执行 `npm test`、`npm run build` 和 `git diff --check`。

## 非目标

- 不删除或替换真实的最后一帧。
- 不改变目标帧率策略、画质参数、音频码率或 Upscayl 设置。
- 不通过静音、补黑帧或重复最后一帧掩盖音视频时长差。
- 不为所有容器差异增加零容差式阻断。
