# v0.7.0 README、第三方软件信息与发布设计

## 目标

为 `v0.7.0` 同步更新中英文 README，补齐当前产品能力和 Godot 4.6 视频工作流；扩充应用“关于”窗口中的主要第三方软件信息，并准确展示许可证与官方链接；通过项目现有的 Windows 发布流水线正式发布 `v0.7.0`。

## 范围

### 双语 README

- `README.md` 与 `README.zh-CN.md` 使用对应的章节结构，并把当前版本统一更新为 `0.7.0`。
- 补齐视频处理工作台：按百分比调整分辨率、Upscayl GPU 超分、低画质损失压缩、OGV 转换，以及面向 Godot 4.6 的导入说明。
- 补齐图片处理和精灵图工作台中的高清化能力，以及 BiRefNet 抠图、VoxCPM 配音和 Stable Audio 音效生成等现有功能。
- 更新视频运行时说明：Upscayl GPU 是 GPU 超分所需组件，FFmpeg/FFprobe 负责媒体编解码和探测，并说明应用内运行包安装与缓存行为。
- 说明“关于”窗口只展示主要第三方组件；完整 npm 依赖以 `package.json` 和锁文件为准。
- 不把项目自身许可证、第三方软件许可证、模型许可证和潜在编解码专利义务混为一谈。

### “关于”窗口

- 继续以 `src/openSourceSoftware.ts` 作为第三方信息的唯一数据源。
- 清单采用精选范围：用户可见运行时、AI 模型或服务、直接生产依赖和关键构建工具，不展开全部传递依赖。
- 每项提供名称、用途、许可证名称、许可证链接和项目主页；界面提供可访问的外部链接。
- 按“桌面与媒体运行时”“AI 模型与服务”“应用库”“构建工具”分类，避免长列表难以浏览。
- 对开放权重或自定义许可证使用其官方名称，不将其笼统描述为 OSI 开源软件。
- 主要条目包括 Upscayl/NCNN Vulkan、FFmpeg/FFprobe、BiRefNet、VoxCPM、Gradio、Stable Audio，以及应用直接使用的核心库和 Electron 构建工具。

## 数据与界面边界

- 第三方条目类型在数据模块中定义，分类和许可证元数据不在 React 组件中重复维护。
- `SiteFooter` 只负责按分类渲染“关于”内容和链接；不会引入依赖扫描、网络请求或许可证推断逻辑。
- README 与“关于”窗口共享术语，但不共享运行时代码；README 面向安装和使用，“关于”窗口面向归属和许可证查询。

## 正确性与合规表述

- 许可证名称和链接以项目或模型的官方仓库、官方网站或官方许可证文件为依据。
- 对 FFmpeg/H.265 仅陈述软件许可证和用户需要自行评估编解码器专利授权，不承诺商业使用必然免费。
- 对模型权重区分代码许可证与模型许可证；无法由官方来源确认的许可证不做推断。
- 外部链接使用安全的新窗口打开属性，并保留可读的链接文字或无障碍标签。

## 测试与验证

- 先更新结构测试，约束第三方条目的许可证字段、关键组件覆盖和“关于”窗口许可证链接渲染。
- 检查双语 README 的版本号、主要功能章节、Godot 4.6 工作流和第三方说明。
- 发布前执行：
  - `npm test`
  - `npm run build`
  - `git diff --check`
- 将 `package.json` 与 `package-lock.json` 的版本更新为 `0.7.0`，并确保仓库无意外改动。

## 发布流程

1. 在 `master` 提交实现并推送到 `origin/master`。
2. 创建并推送注解标签 `v0.7.0`。
3. 从 `master` 手动触发 `.github/workflows/windows-release.yml`，传入 `tag=v0.7.0`。
4. 等待 GitHub Actions 成功完成。
5. 核对版本化 Release `v0.7.0`、稳定通道 Release `windows-x64-latest` 及其 `latest.yml`；三者都必须指向 `0.7.0`。
6. 如果测试、构建、推送或发布工作流任一步骤失败，停止并修复，不上传未经验证的本地构建产物。

## 非目标

- 不在“关于”窗口列出所有 npm 传递依赖。
- 不实现自动许可证扫描器或生成独立法律意见。
- 不改变现有工作台行为或发布渠道结构。
- 不承诺任何第三方编码格式、模型或软件在所有地区和商业场景下均无额外授权要求。
